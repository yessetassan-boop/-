# Конспект: ООП-база — сессия 1

Формат: коротко, для быстрого recall перед собесом.

---

## 1. Interface vs Abstract class

- **interface** = контракт поведения. Без состояния. Члены `public`. Конструктора нет.
- **abstract class** = частичная реализация + состояние.

**Что abstract умеет, а interface — нет:**
- поля (изменяемое состояние) + конструктор;
- общая реализация `equals` / `hashCode` / `toString`;
- `protected` / package-private члены = непубличные точки расширения.

**Ещё:**
- класс реализует МНОГО интерфейсов, наследует ОДИН класс;
- с Java 8 interface умеет `default` / `static` методы → «общая реализация» уже не эксклюзив abstract;
- поля в interface = только `public static final` константы (не состояние).

**Собес одной строкой:** есть общее состояние → abstract. Нет → interface.

---

## 2. Template Method (паттерн)

- публичный скелет алгоритма → `final` (защищает **инвариант**);
- варьируемый шаг → `protected abstract` (точка расширения);
- общие шаги → `private` (заперты, не протекают в наследников).

**Формула:** `final` + `protected abstract` = защита инварианта при контролируемой вариативности.

**Главное — правильно мапить:** что одинаково → в базу (заперто); что отличается → `abstract` (наследникам).

## 3. Полиморфизм: overload vs override

**Две фазы:**
- **компайл-тайм** — `javac` читает текст, знает только объявленные (статические) типы;
- **рантайм** — JVM исполняет, знает реальные объекты.

| | разрешается | по чему |
|---|---|---|
| **overload** (перегрузка) | компилятор | статический тип выражения |
| **override** (переопределение) | JVM | реальный тип объекта |

- override требует **совпадения сигнатуры** (имя + типы параметров);
- ставь `@Override` — компилятор поймает несовпадение;
- каст меняет статический тип выражения → меняет выбор перегрузки.

---

## 4. Полиморфизм: что НЕ полиморфно

- **Полиморфны:** обычные переопределённые instance-методы (по реальному объекту).
- **НЕ полиморфны** (по типу ссылки, это *hiding*): **поля**, **`static`-методы**, **`private`-методы**.

**Поля:** объект держит ОБА поля (родителя и наследника) одновременно; какое видишь — зависит от типа ссылки.

**На практике:** поля `private` + доступ через геттер (метод → полиморфен) → ловушка исчезает сама.

## 5. equals / hashCode

- работают **в паре**, нужны хеш-коллекциям (`HashMap` / `HashSet`).

**Механизм поиска:**
1. `hashCode` → номер ящика (bucket);
2. внутри ящика → сравнение через `equals`.

**Правило:** переопределил `equals` → ОБЯЗАН переопределить `hashCode`, из тех же полей.

**Контракт:**
- `a.equals(b)` → hashCode **обязан** совпадать;
- одинаковый hashCode → объекты **НЕ обязаны** быть равны (коллизии ок).

**Баг:** `equals` без `hashCode` → равные объекты в разных ящиках → `contains` = `false` (а `equals` даже не вызовется).

**Fix:** `return Objects.hash(x, y);`

---

## 6. equals: контракт (5 правил)

рефлексивность · симметрия · транзитивность · консистентность · `x.equals(null)` = false

**Ловушка симметрии на наследовании** (Point / ColorPoint):
- `instanceof` однонаправлен: ColorPoint **is-a** Point, но Point **не is-a** ColorPoint;
- `p.equals(cp)` = `true`, а `cp.equals(p)` = `false` → симметрия разбита.

**Вывод:** нельзя расширить конкретный класс новым полем-значением и сохранить `equals` с `instanceof`.

---

## 7. Композиция vs наследование

- **композиция** = `has-a` (держит объект как поле, делегирует ему);
- **наследование** = `is-a`.
- **DI в Spring = композиция** (сервис держит репозиторий, а не наследует его).

**Почему composition > inheritance:** слабая связь, не тянет чужую иерархию, не ломает инкапсуляцию.

**Фикс ColorPoint:** держит `Point` внутри (не наследует) → симметрия цела (оба `false`).

**Мораль на собес:** `equals` ломается на наследовании → замени `is-a` на `has-a`.
